## 2. "Investors Top Choice" — синтетический портфель по консенсусу

**Motivation.** Гипотеза: «если суперинвесторы вкладывают больше денег в
актив, они в него больше верят». Значит, портфель из топ-N активов с
максимальным консенсус-весом — это валидируемая стратегия. Хочется проверить,
насколько такой портфель обыгрывает VOO / VT, и сравнить его с конкретными
инвесторами.

Логика для каждого квартала уже есть в `ConsensusPanel` (`:1311`) — она для
каждого `viewMode='held'` агрегирует веса по всем `includedPortfolios`,
сортирует по `combined = total / N`. Нужно сделать то же самое **поквартально
по всей истории** и собрать из этих снапшотов настоящий портфель с `history[]`,
который пройдёт через `computeSeries` точно так же, как любой guru.

**Sketch (для обсуждения).**

- На каждый `asOf` в `meta.priceYears`-окне берём все портфели типа `guru` из
  `selectedInvestors` (или из всех — открытый вопрос ниже), агрегируем веса по
  тикеру, отрезаем по топ-N (5/10/15/20), нормализуем на 100. Это даёт
  `history[].holdings` для синтетика.
- Для текущего квартала — то же, но из `holdings`, и кладём в `holdings`
  результат.
- В index добавляем виртуальные id типа `top-choice-5`, `top-choice-10`,
  `top-choice-15`, `top-choice-20`. `kind: 'synthetic'`. `usesFilingLag` для
  них должен возвращать `true` (это всё ещё 13F-консенсус, доступный только
  через 45 дней после квартала).
- Не писать эти портфели на диск — генерировать в рантайме при загрузке.

**Open questions (важные, нужно решить до начала).**

1. **Один профиль с переключателем top-N, или 4 отдельных профиля?**
   Аргумент за один: чище в списке портфелей, меньше шума на графике.
   Аргумент за четыре: можно одновременно сравнить линии «top-5 vs top-20»,
   что и есть суть гипотезы.
2. **Из кого считать консенсус?** Из `selectedInvestors` (то, что юзер выбрал)
   или из всех 81 инвесторов из базы? Если из selected — синтетик меняется
   при добавлении / удалении инвестора из списка, что нелогично для
   «эталонного» индикатора. Если из всех — нужно подгружать все 81 файлов на
   старте (сейчас лениво по `selectedInvestors`). Скорее всего «из всех», но
   тогда нужен флаг в `meta.json` типа `topChoiceUniverse: 'all' | 'curated'`.
3. **Шарпай ли веса.** Брать долю каждого инвестора в его собственном
   портфеле напрямую (как сейчас в `held` mode) или нормализовать по числу
   инвесторов, владеющих тикером (more holders == higher weight)? Первый
   вариант ближе к идее «больше денег = больше веры», второй — ближе к
   «consensus pick». Стоит реализовать оба и сравнить.
4. **Что считать «верой» при разном AUM.** Berkshire с $300B и Vinall с
   $500M вносят разный реальный капитал. Сейчас веса от 0 до 100 у обоих
   нормализованы — это «доля портфеля», а не «доля денег». Надо ли AUM-
   взвешивать? Сейчас `aum` у большинства инвесторов `null` в
   `investors-index.json` — без AUM-данных эта оптимизация невозможна, но
   стоит держать в голове.

**Touches.**

- Новый файл / функция (например `src/topChoice.js` или прямо в трекере)
  с детерминированным генератором `buildTopChoicePortfolio(allInvestors, N)`.
- `src/portfolio_tracker.jsx:3097` `fetchDefaultData` — генерация после
  загрузки `investorsIndex`.
- `src/portfolio_tracker.jsx:167` `usesFilingLag` — добавить
  `kind === 'synthetic'`.
- Логика уже отлажена в `ConsensusPanel:1421` `stats` useMemo — её можно
  переиспользовать.

## 3. Eye-toggle глазика должен пересчитывать матрицу

**Motivation.** Сейчас в модалке портфеля (`PortfolioEditModal:590`) можно
выключить отдельный тикер у инвестора через «глазик»
(`disabledHoldings` `:3006`, `getActiveHoldings` `:110`). При выключении актив
**исключается из расчётов в реальном времени для текущего квартала и для
истории — как будто инвестор им никогда не владел**. Это работает на графике
и в Consensus, но **не** в `InvestorsMatrix` и `ETFsMatrix`.

**Sketch.**

- Прокинуть `disabledHoldings` в `InvestorsMatrix` (`:1965`) и `ETFsMatrix`
  (`:2510`).
- В `rawSeriesByInvestor` useMemo (`:2027`) перед вызовом `computeSeries`
  применять `getActiveHoldings(p, disabledHoldings)` не только к `holdings`,
  но и ко всем `history[].holdings` — фильтр должен ретроспективно вычистить
  тикер из всех квартальных снапшотов. После фильтрации **обязательно**
  перенормировать веса на 100, иначе `computeSeries` будет считать с
  «дыркой» в портфеле.
- Зависимость `disabledHoldings` добавить в useMemo deps.

**Open questions.**

- Перенормировка после удаления тикера сейчас в `getActiveHoldings` **не**
  делается — она просто фильтрует. На графике это работает потому, что

  каждого segment'а — проверить, что ретроспективное отключение не ломает
  chain-link continuity (cumulative carry на границах сегментов).

**Touches.**

- `src/portfolio_tracker.jsx:2027` `rawSeriesByInvestor` — добавить применение
  фильтра.

## 4. Переезд парсинга данных на GitHub Actions

**Motivation.** Сейчас обновление инвесторов и цен идёт через
`/goal` + LLM (см. `goals/INVESTORS-BACKFILL.md`, `goals/STOCKS-UPDATE.md`).
**LLM могут ошибиться**, а точность цифр в этом приложении — критична: если
кумулятивная return-кривая инвестора расходится с реальной на 2–3% из-за
того что LLM подцепил неверный тикер или пропустил квартал, все выводы
бессмысленны.

GitHub Actions с детерминированным Python/Node-скриптом, парсящим
DataRoma / 13F.info напрямую, — куда надёжнее. LLM остаётся как «человек на
ручке», который правит крайние случаи (новые инвесторы, ребрендинг тикеров),
а не как ежеквартальный парсер.

**Sketch.**

- Workflow `.github/workflows/stocks-update.yml` — cron каждое 1 число месяца:
  - Скрипт на Node, читающий `investors-index.json`, собирающий уникальный
    список тикеров, дёргающий Yahoo Finance (или Alpha Vantage) монтлли,
    обновляющий `public/data/prices/<YYYY>.json`.
- Workflow `.github/workflows/investors-backfill.yml` — cron 16-го числа
  каждого 3-го месяца (15 / Aug 15 / Nov 15 / Feb 15 — день после дедлайна
  13F):
  - Парсит DataRoma по `sourceCodes.dataroma` каждого инвестора, делает
    chain-link обновление как в текущем `goals/INVESTORS-BACKFILL.md`, но
    детерминированно. Падает с диагностикой при любом расхождении.
- Action коммитит результат на отдельную ветку `data/<timestamp>` и открывает
  PR. PR — единственный путь данных в `main`.
- **Контракт приоритета.** Данные, попавшие в `main` через Action-PR, —
  source of truth. Любой LLM (через `/goal` или вручную) **не имеет права**
  переписывать тикер / `shares` / `weight` из Action-PR без явного указания
  пользователя. LLM может только дополнять (добавить нового инвестора,
  починить новый corner case), но не «переоткрывать» цифры.
- После переезда — удалить `goals/STOCKS-UPDATE.md` и
  `goals/INVESTORS-BACKFILL.md`. `goals/INVESTORS-ADD.md` остаётся, потому
  что добавление нового инвестора — это редкая ad-hoc операция, для которой
  LLM-навигация по разным источникам ценнее автоматизации.

**Open questions.**

- Какой бесплатный источник цен с приемлемым rate limit для месячного
  кронa? Yahoo Finance via `yfinance` работает, но официально не
  поддерживает HTTP API — могут сломать в любой момент. Alpha Vantage —
  жёсткий лимит 25 запросов в день в free tier. Stooq, EOD Historical Data
  — кандидаты для оценки.
- DataRoma структурирован в HTML, парсится тривиально. 13F.info — таблицы,
  тоже ок. Что делать с инвесторами, для которых `primarySource` —
  не-DataRoma? Скорее всего, Action поддерживает только DataRoma; для
  остальных оставить `/goal`.
- Где хранить секрет API-ключа (если будет нужен) — GitHub Secrets.
- Что делать, если DataRoma внезапно недоступна — Action падает, PR не
  открывается, в `meta.json` `lastBackfillAt` не обновляется. Юзер видит
  это в UI (timestamp в шапке).

**Touches.**

- Новые файлы `.github/workflows/*.yml`.
- Новый каталог `scripts/ci/` с детерминированными парсерами.
- `goals/README.md` — обновить таблицу goals, явно сказать, что
  `STOCKS-UPDATE` и `INVESTORS-BACKFILL` теперь автоматические.
- `CLAUDE.md` — добавить раздел «Action-PR данные имеют приоритет над LLM».

## 5. ETF — реальный состав поквартально (декоративно)

**Motivation.** Сейчас ETF в `public/data/etfs-index.json` — это просто
метаданные (ticker, name, issuer). В `fetchDefaultData` (`:3170`) он
превращается в портфель с `holdings: [{ticker: meta.ticker, weight: 100}]`,
то есть «100% самого себя». Это работает математически (цена ETF тянется
напрямую из `prices`), но **скрывает** информацию о том, что внутри ETF.

Хочется показывать в модалке ETF или в матрице, **что реально содержится**
в фонде сейчас и что содержалось в прошлых кварталах — для понимания
«это тематический ETF, основанный на NVIDIA и AMD» vs «диверсифицированный
по 500 компаниям».

**Sketch.**

- В `public/data/etfs-index.json` для каждого ETF добавить опциональное
  поле `decorativeHoldings[]` — структура такая же, как у инвестора
  (`{ticker, weight}[]`), плюс `decorativeHistory[]` с `asOf`.
- **Эти данные — чисто визуальные.** `computeSeries` для ETF продолжает
  использовать одноэлементный `holdings: [{ticker: meta.ticker, weight: 100}]`
  — то есть цена строится **только** из `prices/<YYYY>.json[meta.ticker]`.
  Никакой математики по составу.
- Модалка ETF (новый компонент или расширение `PortfolioEditModal`) показывает
  decorative-состав как read-only список с переключателем кварталов.
- Эту информацию парсит тот же GitHub Action из пункта 4 (там, где есть
  открытый источник — Vanguard / iShares публикуют составы).

**Open questions.**

- Где брать decorative-состав. SSGA / Vanguard / iShares публикуют PDF /
  Excel-таблицы; нет единого API. Возможно, начать с пары хорошо
  документированных (VOO через S&P 500, QQQ через Nasdaq-100) и расширять.
- Когда decorative-состав отсутствует — модалка ETF просто не показывает
  раздел «Holdings», без ошибок.
- BTC-USD / GLD — не имеют holdings концептуально. Должны корректно
  показывать только цену.

**Touches.**

- `public/data/etfs-index.json` — расширение схемы.
- `src/portfolio_tracker.jsx:3170` `etfPortfolios` — НЕ менять
  (`holdings` остаётся single-ticker, иначе ломается чистота расчётов).
- Новый UI-слой для отображения `decorativeHoldings` — отдельная
  модалка / секция в существующей.

## 6. Фильтр по тикерам в объединённой матрице

**Motivation.** После объединения Investors+ETF (пункт 1) появляется
естественный use case: «покажи всех (инвесторов и ETF), у кого в портфеле
есть GOOGL». Это инверсия consensus — не «что у всех есть», а «кто разделяет
мою ставку на конкретный тикер».

Use case: «я понимаю Google, не понимаю энергетику; хочу видеть только тех,
кто инвестирует в Google так же, как и я».

**Sketch.**

- Сверху матрицы — search input + чипы выбранных тикеров.
- Фильтр пропускает строку, если в её `holdings` (или `history[].holdings`,
  см. open question) есть **любой** из выбранных тикеров. Если выбрано
  несколько — `AND` (все) или `OR` (любой) — см. open question.
- Дополнительно — показать в каждой строке мини-индикатор веса тикера
  у этого инвестора (например крошечную колонку справа от названия).

**Open questions.**

1. **«Сейчас держит» или «когда-либо держал»?** Первое прозрачнее,
   второе шире. Возможно решение — чип-фильтр с двумя режимами или
   небольшой переключатель `Currently / Ever`.
2. **OR vs AND для нескольких тикеров.** Обе семантики полезны: OR =
   «кто-то из этих»; AND = «у кого вся моя корзина целиком». Возможно
   простой `mode: 'any' | 'all'` рядом с инпутом.
3. **Что считать тикером.** С учётом `mergeMode` (BRK.A/BRK.B → BRK),
   нормализация ввода через `normalizeTicker` (`:58`).

**Touches.**

- Новый useState в `InvestorsMatrix` (после переезда на единую матрицу из
  пункта 1).
- Применение фильтра поверх `sortedInvestors` (`:2158`).

## 7. VT / VOO как глобальный toggle вместо обычных портфелей

**Motivation.** Сейчас `VOO` и `VT` — это обычные `benchmark`-портфели в
`public/default-data.json`. Их можно скрыть / показать / удалить как любой
другой портфель, что нелогично — это **референсные** инструменты, без
которых страница теряет смысл («с чем мы вообще сравниваем?»).

В `Investors`-матрице есть переключатель `Absolute | vs VOO | vs VT`
(`:1970` `mode`) — там это уже решено через mode-tabs.

**Sketch.**

- В шапке Portfolio-вкладки (рядом с UI scale / Theme / Save) — чекбокс
  «Show market benchmarks (VOO, VT)». По умолчанию ON.
- Когда ON — VOO и VT всегда **вверху** списка портфелей, всегда видимы как
  линии на графике (или хотя бы по умолчанию visible), не управляются
  drag-and-drop reorder, не удаляются.
- Когда OFF — обе линии скрыты с графика и убраны из списка.
- В `default-data.json` хранится только сам флаг
  (`showMarketBenchmarks: boolean`) — сами портфели генерируются из
  `etfs-index.json` (там VOO и VT уже есть как `etf-voo`, `etf-vt`).

**Open questions.**

- Сейчас в `default-data.json` есть верхний массив `benchmarks: [...]` с
  явно прописанными VOO и VT. После миграции этот массив **полностью
  удаляется** (no legacy fallback per `CLAUDE.md`). Скрипт миграции —
  одноразовый, в `scripts/`, удаляется сразу после применения.
- `chartMode` (`'absolute' | 'voo' | 'vt'`) и кнопки «vs VOO» в шапке
  графика — остаются. Это другой механизм (математический режим, не
